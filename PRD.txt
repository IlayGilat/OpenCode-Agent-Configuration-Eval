# TypeScript Local Agent Evaluation System Report

## 1. Detailed Design

### 1.1 Purpose

The purpose of this system is to create a simple local evaluation harness for testing how well the current OpenCode agent setup performs on real historical Jira tickets from one local repository.

This is not a production benchmark system.
This is not a cloud service.
This is not a CI/CD system.
This is not a generic multi-repo platform.

It is a Windows-only local TypeScript CLI application that loops through around 50 Jira tickets, lets OpenCode solve each ticket, compares the generated patch against the real merged PR patch, and produces a final report with:

* median score
* average score
* score table
* verdict distribution
* executive summary
* common strengths
* common problems
* recommendations for improving `AGENTS.md`, OpenCode agents, skills, MCPs, and repo instructions

The main thing being evaluated is not a different model.
The main thing being evaluated is the current local agent setup around the model:

```text
AGENTS.md
OpenCode config
agents
subagents
skills
MCPs
repo conventions
prompting setup
```

The same model is used for solving and judging.

---

### 1.2 Main Evaluation Idea

For every Jira ticket, the system does the following:

```text
1. Checkout the repo at the commit before the original PR.
2. Give OpenCode only the Jira title and Jira description.
3. Let OpenCode inspect the repo and make changes.
4. Capture the resulting git diff as candidate.patch.
5. Generate the real PR patch from baseCommit to goldCommit.
6. Ask OpenCode, acting as judge, to compare candidate.patch to gold.patch.
7. Save a score from 0 to 100.
8. Continue to the next ticket.
9. At the end, aggregate all scores and generate reports.
```

The solver does not see the real PR, gold patch, judge prompt, or scoring rubric.

The judge does see:

```text
ticket.md
candidate.patch
gold.patch
rubric
```

This is subjective because there are no automated tests.
The goal is not perfect correctness.
The goal is to estimate whether the generated PR “looks good” compared to the real merged PR.

### 1.3 Key Constraints

The design follows these constraints:

```text
- Windows only
- Local only
- One repo only
- Around 50 Jira tickets
- No automated tests
- No Jira MCP
- No database
- No web server
- No Docker
- No CI/CD
- No package version pinning in the design document
- One model used for both solver and judge
- OpenCode is run headlessly
- Filesystem is used as storage
- The application is written in TypeScript
- The system should not be a god file
```

### 1.5 Windows-Only Rules

Because the app only needs to run on Windows, the implementation can be optimized for local PowerShell usage.

However, path handling should still be clean.

Use Node’s path utilities:

```ts
import path from "node:path";

path.resolve(...);
path.join(...);
```

Do not manually concatenate paths:

```ts
const filePath = basePath + "/" + fileName;
```

This is especially important on Windows because paths may look like:

```text
C:\Users\you\projects\repo
```

or may be written in config as:

```json
{
  "repoPath": "../my-real-repo",
  "workspacePath": "./workspace"
}
```

The config can use forward slashes.
The application should normalize them internally with `path.resolve`.

Cleanup should be done in TypeScript, not shell commands.

Use:

```ts
await fs.rm(workspacePath, {
  recursive: true,
  force: true,
});
```

Do not rely on Linux/macOS commands like:

```bash
rm -rf workspace
```

Shelling out should be limited to:

```text
git
opencode
```

Everything else should be done in TypeScript.

### 1.7 Input Config

The system uses one config file.

`eval.config.json`

```json
{
  "repoPath": "../my-real-repo",
  "workspacePath": "./workspace",
  "runsPath": "./runs",
  "reportsPath": "./reports",
  "opencodeCommand": "opencode",
  "model": "your-provider/your-model",
  "timeoutMinutes": 60
}
```

Field meaning:

| Field             | Meaning                                              |
| ----------------- | ---------------------------------------------------- |
| `repoPath`        | Path to the real local Windows repo with git history |
| `workspacePath`   | Local folder where clean worktrees are created       |
| `runsPath`        | Local folder where per-ticket artifacts are saved    |
| `reportsPath`     | Local folder where final reports are written         |
| `opencodeCommand` | Usually `opencode`                                   |
| `model`           | Same model used for solver and judge                 |
| `timeoutMinutes`  | Maximum time for each OpenCode call                  |

There is no separate `solverModel` and `judgeModel`.

Use one field:

```json
{
  "model": "your-provider/your-model"
}
```

The same model is used twice:

```text
solver phase
judge phase
```

### 1.8 Tickets Input

All Jira tickets are stored in one local JSON file.

`tickets.json`

```json
[
  {
    "id": "JIRA-001",
    "title": "Fix settings modal not saving theme",
    "description": "When the user changes the theme in the settings modal and clicks save, the modal closes but the theme is not persisted.",
    "baseCommit": "abc123",
    "goldCommit": "def456"
  },
  {
    "id": "JIRA-002",
    "title": "Fix export crash when item list is empty",
    "description": "The export flow crashes when the selected item list is empty. It should export an empty file instead.",
    "baseCommit": "aaa111",
    "goldCommit": "bbb222"
  }
]
```

Each ticket requires:

| Field         | Meaning                                       |
| ------------- | --------------------------------------------- |
| `id`          | Stable ticket ID used for folders and reports |
| `title`       | Jira ticket title                             |
| `description` | Jira ticket description                       |
| `baseCommit`  | Commit before the original PR work started    |
| `goldCommit`  | Commit after the real solution was merged     |

The system assumes `repoPath` contains the full git history required for all base and gold commits.

### 1.10 Storage Model

The system uses the filesystem as the database.

There are three main storage areas:

```text
workspace/
runs/
reports/
```

#### `workspace/`

Temporary git worktrees are created here.

Example:

```text
workspace/
  JIRA-001/
  JIRA-002/
```

Each workspace is a checkout of the real repo at the ticket’s `baseCommit`.

The solver runs inside this workspace.

#### `runs/`

Permanent per-ticket artifacts are saved here.

Example:

```text
runs/
  JIRA-001/
    ticket.md
    gold.patch
    candidate.patch
    solver-output.txt
    judge-input.md
    judge-output.txt
    score.json
```

This folder is useful for debugging and auditability.

#### `reports/`

Final summary outputs are saved here.

Example:

```text
reports/
  results.json
  results.csv
  summary.md
```

### 1.14 Per-Ticket Execution Flow

For each ticket, `RunService.runTicket(ticket)` should do this:

```text
1. Create runs/<ticketId>/.
2. Prepare workspace/<ticketId>/ at baseCommit.
3. Write runs/<ticketId>/ticket.md.
4. Generate gold.patch using git diff baseCommit goldCommit.
5. Run OpenCode solver in workspace/<ticketId>/.
6. Save solver-output.txt.
7. Capture candidate.patch using git diff.
8. Save candidate.patch.
9. If candidate.patch is empty, create automatic fail score.
10. Build judge-input.md.
11. Run OpenCode judge.
12. Save judge-output.txt.
13. Parse judge JSON.
14. Save score.json.
15. Return JudgeResult.
```

The workspace should be created from the original repo using git worktree.

Example git command:

```text
git -C <repoPath> worktree add --detach <worktreePath> <baseCommit>
```

The gold patch should be generated from the original repo:

```text
git -C <repoPath> diff <baseCommit> <goldCommit>
```

The candidate patch should be generated from the solver workspace:

```text
git -C <workspacePath> diff
```

### 1.15 Solver Prompt

The solver prompt should be simple and consistent.

```md
You are working in a real codebase.

Implement the Jira ticket below.

Rules:
- Read the repo freely.
- Follow AGENTS.md and repo conventions.
- Use existing patterns.
- Make the smallest reasonable change.
- Do not rewrite unrelated code.
- Do not access the internet.
- At the end, summarize what changed.

Ticket:

# <JIRA_ID>: <TITLE>

<DESCRIPTION>
```

The solver should receive only:

```text
Jira title
Jira description
repo at baseCommit
current AGENTS.md / OpenCode config / skills / MCPs
```

The solver should not receive:

```text
gold.patch
goldCommit
judge prompt
score rubric
previous scores
previous candidate patches
```

This is important because the evaluation is supposed to test whether the current agent setup can solve the ticket from the same kind of information a real agent would receive.

---

### 1.16 Candidate Patch

After the solver finishes, the system captures the generated diff.

```text
git -C <workspacePath> diff
```

The output is saved as:

```text
runs/<ticketId>/candidate.patch
```

This file represents the AI-generated PR.

If the candidate patch is empty, the system should avoid running the judge and create an automatic fail score.

Automatic empty patch score:

```json
{
  "score": 0,
  "verdict": "fail",
  "solve_probability": 0,
  "gold_alignment": 0,
  "repo_pattern_quality": 0,
  "minimality": 0,
  "risk": 0,
  "would_i_merge": false,
  "summary": "The solver produced no code changes.",
  "strengths": [],
  "problems": ["No candidate patch was produced."]
}
```

---

### 1.17 Gold Patch

The gold patch is the real merged PR diff.

It is generated using:

```text
git -C <repoPath> diff <baseCommit> <goldCommit>
```

The output is saved as:

```text
runs/<ticketId>/gold.patch
```

The solver must not see this patch.

The judge uses it as reference.

The candidate does not need to match the gold patch exactly.
The gold patch is used to understand the real intent, scope, and likely missing logic.

---

### 1.18 Judge Prompt

The judge prompt should be strict and should request JSON only.

```md
You are judging an AI-generated code change.

You will receive:

1. Jira ticket
2. Gold/reference patch from the real merged PR
3. Candidate patch created by an AI agent

Important:
The candidate does not need to match the gold patch exactly.
Different code is acceptable if it likely solves the same ticket.
Use the gold patch as a reference for intent, scope, and likely missing pieces.

Score the candidate from 0 to 100.

Rubric:
- 50 points: high chance of solving the ticket
- 20 points: conceptually aligned with the gold PR
- 15 points: follows existing repo patterns
- 10 points: minimal and focused change
- 5 points: low obvious regression risk

Return only valid JSON:

{
  "score": 0,
  "verdict": "fail | weak | partial | good | excellent",
  "solve_probability": 0,
  "gold_alignment": 0,
  "repo_pattern_quality": 0,
  "minimality": 0,
  "risk": 0,
  "would_i_merge": false,
  "summary": "",
  "strengths": [],
  "problems": []
}
```

Recommended interpretation:

| Score Range | Verdict     |
| ----------: | ----------- |
|      `0-24` | `fail`      |
|     `25-49` | `weak`      |
|     `50-69` | `partial`   |
|     `70-84` | `good`      |
|    `85-100` | `excellent` |

---

```md
The current agent setup scored a median of 76 across 50 evaluated Jira tickets.

## Median Score

76

## Score Table

| Ticket | Score | Verdict | Merge? |
|---|---:|---|---|
| JIRA-001 | 78 | good | yes |
| JIRA-002 | 64 | partial | no |
| JIRA-003 | 87 | excellent | yes |

## Verdict Distribution

| Verdict | Count |
|---|---:|
| Excellent | 8 |
| Good | 17 |
| Partial | 14 |
| Weak | 7 |
| Fail | 4 |

## Common Strengths

1. Finds the obvious implementation area.
2. Usually follows local code style.
3. Produces focused patches for simple tickets.
4. Often makes changes that are conceptually close to the real PR.

## Common Problems

1. Misses shared helper functions.
2. Makes a local fix instead of fixing the root pattern.
3. Does not always notice related files.
4. Sometimes implements the obvious behavior but misses edge cases from the real PR.
5. Occasionally touches unrelated files.

Improve AGENTS.md with:

1. Important shared utilities and where they live.
2. Common implementation layers.
3. Feature ownership patterns.
4. Files or folders that should usually not be edited.
5. A final checklist before finishing a ticket.
```

### 1.21 Median Score Logic

The main score should be the median, not the average.

Median is better here because:

```text
- One terrible run should not destroy the entire evaluation.
- One excellent run should not make the setup look better than it is.
- The benchmark is subjective and judge-based.
- The score distribution may be noisy.
```

Median implementation:

```ts
function median(values: number[]): number {
  if (values.length === 0) {
    return 0;
  }

  const sorted = [...values].sort((a, b) => a - b);
  const middle = Math.floor(sorted.length / 2);

  if (sorted.length % 2 === 1) {
    return sorted[middle];
  }

  return (sorted[middle - 1] + sorted[middle]) / 2;
}
```

### 1.23 Main Service Pseudocode

`RunService` is the central orchestrator.

It should coordinate the flow, but not implement every detail itself.

```ts
export class RunService {
  constructor(
    private readonly workspaceService: WorkspaceService,
    private readonly patchService: PatchService,
    private readonly solverRunner: SolverRunner,
    private readonly judgePromptBuilder: JudgePromptBuilder,
    private readonly judgeRunner: JudgeRunner,
    private readonly judgeResultParser: JudgeResultParser,
    private readonly runRepository: RunRepository,
  ) {}

  async runTicket(ticket: JiraTicket): Promise<JudgeResult> {
    await this.runRepository.createRunFolder(ticket.id);

    const workspacePath = await this.workspaceService.prepare(ticket);

    await this.runRepository.writeTicketMarkdown(ticket);

    const goldPatch = await this.patchService.createGoldPatch(ticket);
    await this.runRepository.writeGoldPatch(ticket.id, goldPatch);

    const solverOutput = await this.solverRunner.solve(ticket, workspacePath);
    await this.runRepository.writeSolverOutput(ticket.id, solverOutput);

    const candidatePatch = await this.patchService.captureCandidatePatch(workspacePath);
    await this.runRepository.writeCandidatePatch(ticket.id, candidatePatch);

    if (this.patchService.isEmptyPatch(candidatePatch)) {
      const emptyScore = createEmptyPatchScore(ticket.id);
      await this.runRepository.writeScore(ticket.id, emptyScore);
      return emptyScore;
    }

    const judgePrompt = this.judgePromptBuilder.build({
      ticket,
      goldPatch,
      candidatePatch,
    });

    await this.runRepository.writeJudgeInput(ticket.id, judgePrompt);

    const rawJudgeOutput = await this.judgeRunner.judge(judgePrompt, workspacePath);
    await this.runRepository.writeJudgeOutput(ticket.id, rawJudgeOutput);

    const score = this.judgeResultParser.parse(ticket.id, rawJudgeOutput);
    await this.runRepository.writeScore(ticket.id, score);

    return score;
  }
}
```

### 1.24 OpenCode Runner Pseudocode

`OpenCodeRunner` is a generic wrapper around `opencode run`.

```ts
export class OpenCodeRunner {
  constructor(
    private readonly config: EvalConfig,
    private readonly processRunner: ProcessRunner,
  ) {}

  async run(input: {
    cwd: string;
    prompt: string;
  }): Promise<string> {
    const result = await this.processRunner.run(
      this.config.opencodeCommand,
      [
        "run",
        "--dir",
        input.cwd,
        "--model",
        this.config.model,
        "--format",
        "text",
        input.prompt,
      ],
      {
        timeoutMs: this.config.timeoutMinutes * 60 * 1000,
      },
    );

    return [
      result.stdout,
      result.stderr ? `\n\nSTDERR:\n${result.stderr}` : "",
    ].join("");
  }
}
```

### 1.25 Git Service Pseudocode

`GitService` wraps git commands.

```ts
export class GitService {
  constructor(private readonly processRunner: ProcessRunner) {}

  async createWorktree(repoPath: string, worktreePath: string, commit: string): Promise<void> {
    await this.processRunner.run("git", [
      "-C",
      repoPath,
      "worktree",
      "add",
      "--detach",
      worktreePath,
      commit,
    ]);
  }

  async removeWorktree(repoPath: string, worktreePath: string): Promise<void> {
    await this.processRunner.run("git", [
      "-C",
      repoPath,
      "worktree",
      "remove",
      "--force",
      worktreePath,
    ]);
  }

  async diff(cwd: string, from?: string, to?: string): Promise<string> {
    const args = ["-C", cwd, "diff"];

    if (from && to) {
      args.push(from, to);
    }

    const result = await this.processRunner.run("git", args);
    return result.stdout;
  }

  async resetHard(cwd: string): Promise<void> {
    await this.processRunner.run("git", ["-C", cwd, "reset", "--hard"]);
  }

  async clean(cwd: string): Promise<void> {
    await this.processRunner.run("git", ["-C", cwd, "clean", "-fd"]);
  }
}
```

### 1.26 Design Rules

#### Rule 1: No God File

Do not put the full system in one `eval.ts`.

Bad:

```text
src/eval.ts
```

Good:

```text
src/config/
src/git/
src/opencode/
src/scoring/
src/reports/
src/runs/
src/tickets/
src/utils/
```

The CLI should parse commands.
`RunService` should orchestrate.
Specialized services should do actual work.

---

#### Rule 2: Solver Does Not See the Gold Patch

The solver receives only:

```text
ticket title
ticket description
repo at baseCommit
current repo-level agent setup
```

The solver does not receive:

```text
goldCommit
gold.patch
candidate comparison
judge rubric
previous scores
```

---

#### Rule 3: Judge Can See Gold Patch

The judge receives:

```text
ticket.md
candidate.patch
gold.patch
rubric
```

The judge uses the gold patch as reference, not as an exact required implementation.

---

#### Rule 4: Empty Patch Is Auto-Fail

If the solver produced no diff, score it immediately as fail.

This saves judge calls and avoids pretending no-op work is meaningful.

---

#### Rule 5: Keep Prompt Stable

The solver prompt and judge prompt should stay the same across all tickets.

This makes the evaluation more consistent.

---

#### Rule 6: Store Everything Useful

For every run, save:

```text
ticket.md
gold.patch
candidate.patch
solver-output.txt
judge-input.md
judge-output.txt
score.json
```

This makes it easy to manually inspect failures later.

---

#### Rule 7: Generate Reports from Saved Scores

The report command should not rerun OpenCode.

It should only read existing `score.json` files and regenerate:

```text
results.json
results.csv
summary.md
```

---

### 1.27 Final Architecture Summary

The system flow is:

```text
CLI
  ↓
ConfigLoader + TicketLoader
  ↓
RunService
  ↓
WorkspaceService + PatchService
  ↓
SolverRunner
  ↓
candidate.patch
  ↓
JudgePromptBuilder
  ↓
JudgeRunner
  ↓
JudgeResultParser
  ↓
score.json
  ↓
ScoreAggregator
  ↓
ReportGenerator
  ↓
summary.md + results.csv + results.json
```

This creates a simple but modular Windows-only TypeScript CLI application.

It avoids a god file while still keeping the system small enough to build quickly.
